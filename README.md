## SCOTTCHEN B2B Website

This repository contains the bilingual SCOTTCHEN B2B lead-generation website built with the Next.js App Router.

### Local development

```bash
npm install
npm run dev
```

Open `http://localhost:3000`.

### Production build

```bash
npm run build
npm run start
```

The site now includes a server-side inquiry endpoint at `/api/inquiry`, so deployment must run on a Node-capable target. The project is configured with `output: "standalone"`.

The `postbuild` step copies `public` and `.next/static` into the standalone bundle. This keeps CSS, fonts, images, the catalog, and metadata routes available when `npm run start` launches the production server.

### Inquiry delivery

Copy `.env.example` to `.env` and configure one delivery path:

- Preferred: `INQUIRY_WEBHOOK_URL`
- Fallback: `RESEND_API_KEY`, `INQUIRY_FROM_EMAIL`, `INQUIRY_TO_EMAIL`
- Optional auto reply: `INQUIRY_AUTO_REPLY_ENABLED=true`, `INQUIRY_AUTO_REPLY_FROM`
- Optional analytics: `NEXT_PUBLIC_GA_MEASUREMENT_ID`

If neither path is configured, the quote and sample forms will show a visible delivery error instead of faking a successful submission.

### Current lead-flow behavior

- Quote and sample forms submit to a server route instead of sending directly from the browser.
- UTM parameters, referrer, locale, and landing page are attached to each submission.
- Form fill start time is attached to each submission for bot filtering.
- Chinese pages keep users inside the Chinese conversion path, including `/zh/thank-you`.
- Shared CTA blocks now route to locale-matched contact and sample pages.
- Personal email addresses are accepted but score lower than company-domain addresses, avoiding unnecessary loss of small-business leads.
- The inquiry endpoint now applies basic in-memory rate limiting and lead scoring.
- First-touch landing page, referrer, and UTM attribution persist while a visitor navigates to the form.
- Google Analytics loads only after explicit visitor consent.
- Light and dark themes follow the visitor's system setting on first load and persist an explicit theme choice locally.

### SEO and content

- Canonical, hreflang, Open Graph, Twitter Card, robots, sitemap, manifest, and organization metadata are generated by the application.
- The English and Chinese resource hubs, supplier-evaluation guides, procurement checklist, and material-selection guide target research-stage B2B search intent and multi-step AI search questions.
- Core product pages include procurement evidence blocks for MOQ, lead time, packaging, QC checks, sample paths, and supplier evaluation links.
- ProductGroup structured data now includes manufacturer references, images, product categories, material notes, and visible sourcing properties.
- Supplier-profile pages document public entity facts and the buyer due-diligence boundary without inventing corporate evidence.
- `/llms.txt` provides a supplemental machine-readable map of core pages and terms that require written confirmation.
- Product photography is delivered as compressed WebP with intrinsic dimensions and deferred loading.
- `public/catalog.pdf` is a generated five-page B2B capability catalog.

To rebuild the catalog:

```bash
UV_NO_PROGRESS=1 uv run --with reportlab --with pillow python scripts/build_catalog.py
```

### Operations

Use `npm run typecheck` for a fast TypeScript pass.

Webhook payloads now include contact fields, inquiry fields, `utm_*`, `referrer`, `landingPage`, `locale`, `clientIp`, `userAgent`, `submittedAt`, `leadScore`, and `leadTier`.

### Cloud Run deployment

`Dockerfile` builds the Next.js standalone server for Cloud Run on port `8080`.
Pushes to `main` are deployed by `.github/workflows/cloud-run.yml` through
Workload Identity Federation, so no long-lived Google Cloud service account key
is stored in GitHub.

Production infrastructure:

- Google Cloud project: `scottchen-b2b-prod-2026`
- Region: `asia-east1`
- Cloud Run service: `scottchen-b2b`
- Artifact Registry repository: `cloud-run`

The canonical host defaults to the deterministic Cloud Run URL. Set
`NEXT_PUBLIC_SITE_URL` to the final HTTPS origin only after the custom domain is
verified, mapped, and serving every route correctly.

The inquiry endpoint requires `INQUIRY_WEBHOOK_URL` or the `RESEND_*` values to
be configured on the Cloud Run service before form submissions can be delivered.
